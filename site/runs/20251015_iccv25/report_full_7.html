ICCV 2025 오픈액세스·코드 공개와 GitHub 생태계 분석

Federlicht assisted and prompted by "Hyun-Jung Kim / AI Governance Team" — 2026-01-27 00:01

제목: ICCV 2025 — 오픈액세스·코드 공개 현황과 GitHub 생태계 심층 리뷰

한 줄 요약: ICCV 2025 논문 대부분이 CVF Open Access 또는 arXiv를 통해 접근 가능하며, GitHub에는 (1) 커뮤니티 집계 레포, (2) 개별 논문별 공식·비공식 구현, (3) 워크숍·데이터셋 레포 등 다층적 저장소가 활발히 존재하지만, 공개 비율·라이선스·재현성에 관한 정량적 검증은 추가 확인이 필요함 (해석).

Lede

ICCV 2025(10월 19–23일, Honolulu)는 전통적으로 그래픽스·컴퓨터비전 커뮤니티가 새로운 방법과 코드 공개 관행을 시험하는 장이었다. 올해도 CVF의 Open Access 포털에 논문별 PDF와 보조자료가 모이고, 동시에 GitHub에서는 여러 형태의 저장소—대규모 집계, 연구실의 공식 코드, 커뮤니티의 재현·튜토리얼 레포—가 병렬로 등장하고 있다 ([ICCV official], [CVF OpenAccess]).

이 보고서는 ‘ICCV 2025’ 키워드에 대한 현장 스냅샷을 제공한다. 공식 일정·오픈액세스 정책을 바탕으로 CVF Open Access와 OpenReview의 공개 항목을 교차검증하고, GitHub 내 대표적 저장소들을 분류·예시 제시하며 향후 검증이 필요한 주요 리스크와 미해결 질문을 정리한다.

Central Question

ICCV 2025에서 발표된 연구는 얼마나 개방적으로 배포되고, GitHub 생태계는 그 배포·재현성 요구를 얼마나 충족하는가?
더 구체적으로, CVF Open Access와 arXiv를 통한 논문 접근성은 어떠하며, GitHub에 올라온 저장소는 ‘공식 코드’·‘비공식 재현’·‘집계’ 중 어떤 분포를 보이는가 — 그리고 이 분포는 연구의 재현성·활용 가능성에 어떤 영향을 미치는가?

The Story So Far

  • 2025년 ICCV 개최·오픈액세스 정책: ICCV 2025는 공식 사이트에 따라 2025년 10월 19–23일에 하와이에서 열리며, CVF는 컨퍼런스 논문의 Open Access(accepted-version PDF 및 보조자료)를 제공한다([ICCV official], [ICCV Dates], [CVF OpenAccess]).
  • 오픈액세스 관행의 확장: CVF Open Access 포털은 ICCV 논문별로 PDF·supplementary·bibtex 링크를 집적해 빠른 공유가 가능하도록 했고, 많은 저자들이 arXiv에도 동일 또는 유사 버전을 업로드해 접근성을 더했다([CVF OpenAccess]).
  • GitHub 생태계의 두 축: 1) 커뮤니티 집계형 레포(예: amusi/ICCV2025-Papers-with-Code)는 논문↔코드 매핑을 빠르게 제공해 ‘탐색 비용’을 낮추는 한편, 2) 개별 연구팀의 공식 레포(hwjiang1510/RayZer, vvangfaye/HoliTracer, NYCU-MAPL/HyTIP 등)는 구현·체크포인트·사용법을 직접 공개해 재현 가능성을 끌어올린다([amusi repo], [RayZer repo], [HoliTracer repo], [HyTIP repo]).
  • 워크숍·챌린지·데이터: OpenAlex와 arXiv 인덱스에는 VQualA, DRL4Real, R-LiViT 등 워크숍·챌린지 보고서가 올라와 부대행사의 데이터·리더보드·코드 링크가 추가적으로 존재함을 보여준다([OpenReview ICCV2025], [OpenAlex works]).
    (요약·연대기 참고: 공식 발표 → CVF OA 업로드 → 저자/커뮤니티의 GitHub 공개 및 집계 레포 생성 순으로 정보가 확산됨.)

How It Works

세 가지 층(layer)으로 생태계를 이해하면 편하다.

H3: 출판·배포 경로 - 저자는 논문을 CVF에 제출·심사 → Acceptance 후 CVF Open Access에 accepted-version PDF가 게시되며(저작권은 저자/권리자에게 유지), 일부는 arXiv에도 업로드한다([CVF OpenAccess]) — 이로써 논문 텍스트와 보조자료가 공개된다.

H3: 코드·데이터 배포 흐름 - 코드 공개 경로는 대체로 세 가지이다: (A) 연구팀 공식 레포(저자·소속기관이 운영), (B) 커뮤니티·학생·연구자가 만든 재현/튜토리얼 레포, (C) 커뮤니티 집계 레포(논문↔코드 인덱싱). 이들 간 신뢰성과 지속성은 매우 다르다([amusi repo], [RayZer repo]). (해석)

H3: 단순화된 메트릭(표현식) - 코드 공개 비율을 샘플 기반으로 측정하려면 다음과 같은 기본 식을 쓸 수 있다: $$\text{code_share} = \frac{N_{\text{papers with linked repos}}}{N_{\text{sampled papers}}}.$$
이 식은 계산이 단순하나, ‘linked repos’의 정의(official? any GitHub link? fork?)와 샘플링 방법에 따라 값이 크게 달라진다(해석).

Why It Matters

  • 과학적 재현성: 논문 텍스트만으로는 재현이 어렵다. 코드·데이터·체크포인트의 공개는 결과 검증과 빠른 후속 연구를 가능하게 한다([CVF OpenAccess], [RayZer repo]). (해석)
  • 접근성과 교육효과: CVF Open Access와 arXiv의 병행 공개는 전 세계 연구자·학생의 접근 장벽을 낮춘다. 이는 산업·학계 간 기술 이전을 가속화한다([CVF OpenAccess]). (전망)
  • 생태계 효율성: GitHub의 집계 레포와 topic 태그(iccv2025)는 새로운 논문을 빠르게 찾아 실험해볼 기회를 준다. 반면, 집계에 의존할 경우 “링크가 오래되었거나 비공식 fork를 가리키는” 문제도 생긴다(해석).
  • 정책·윤리 영향: 공개 정책(저작권·라이선스)은 재사용 범위를 결정한다. CVF OA는 논문 텍스트를 공개하지만 코드·데이터는 별도 라이선스·제공 여부에 의존한다([CVF OpenAccess]). (해석)

(주요 근거: CVF Open Access 포털과 예시 저장소들 — [CVF OpenAccess], [amusi repo], [RayZer repo].)

Open Questions

  1. 정확한 ‘코드 공개 비율’은 얼마인가? (샘플링·정의에 따라 크게 달라짐 — 불확실)
  2. 공개된 코드의 라이선스·사용조건은 얼마나 일관적인가(연구용·상업용 허용 여부)? (불확실)
  3. 커뮤니티 집계 레포(예: amusi)의 커버리지는 공식 CVF 목록과 얼마나 일치하는가 — 오탐·누락 비율은? (불확실)
  4. 장기적 유지보수: ICCV 논문 레포의 ‘영속성’(체크포인트·데모 링크의 계속 작동)은 어떻게 보장되는가? (불확실)
  5. 재현성 검증을 위해 어떤 표준 템플릿(런북, 환경 스냅샷, 체크포인트 지침)이 가장 효율적인가? (제안)

각 질문에는 명시적 샘플링·검증 작업이 필요하다(제안).

Risks & Gaps

  • 데이터/증거의 불완전성: 본 리뷰는 CVF Open Access와 공개된 GitHub 레포(커뮤니티 집계·예시 저장소)를 근거로 한 ‘스냅샷’이다. 전체 논문 집합을 대상으로 한 자동화된 크롤·정량 분석은 수행되지 않았다(근거 부족). 따라서 공개 비율·정확한 수치는 추가 검증이 필요하다.
  • 정의의 불명확성: '코드가 공개되었다'는 주장은 ‘공식 레포’인지, 단순한 노트·스크립트인지, 또는 재현 가능한 형태인지(체크포인트·환경·README 포함)인지에 따라 의미가 달라진다. 본 리뷰에서는 이를 구분해 표기했으나, 표준화된 판정 기준이 필요하다(제안).
  • 라이선스·법적 제약: CVF OA는 논문 텍스트의 공개를 보장하지만 코드·데이터의 재사용은 각 레포의 라이선스에 따르므로, '열린 연구'라고 단정하기 어렵다(근거: CVF OpenAccess 정책과 각 GitHub 레포의 LICENSE 파일).
  • 커뮤니티 집계 신뢰성: amusi 등 집계 레포는 빠르지만 수동 유지·PR 기반이라 누락·중복·오류 가능성이 있다. 자동화된 크롤링 + 원문(논문) 내부의 코드 링크 크로스체크가 필요하다(제안).
  • 검증 필요사항: 정확한 코호트(모든 ICCV 논문 vs. 샘플) 선정, ‘코드 유무’ 판정 규칙, 라이선스 분류 표준을 먼저 정한 뒤 자동화 스크립트로 재현 가능한 통계를 내야 한다(제안).

Critics

짧은 헤드라인: “공개는 많지만, 실제로 믿을 수 있는 공개는 적다” — 반대 시각과 쟁점

비판적 요지: 일부 평론가는 GitHub 중심의 ‘공개 문화’가 과장되어 왔다고 지적한다. 레포가 있어도 환경·데이터·체크포인트가 없으면 재현은 불가능하다는 주장이다(반대 의견).
- 코드의 품질과 문서화가 부족한 경우가 많다; README만 있고 실험 재현에 필요한 스크립트·시드·데이터 전처리 설명이 없다면 공개의 가치는 떨어진다.
- 커뮤니티 집계 레포는 ‘탐색성’을 높이나, 자동화된 스크레이핑으로 인해 링크가 잘못 연결되거나 오래된 fork를 가리키는 문제를 양산할 수 있다.
- 일부는 “오픈액세스 논문 + 코드 공개”를 이상적 조합으로 보나, 실제로는 라이선스 제약·데이터 사용 조건 때문에 산업계·연구자가 자유롭게 활용하지 못하는 경우가 빈번하다고 지적한다.

대안적 권고(요약): - 논문 제출 시 코드·데이터 공개 계획(예정 여부·라이선스)을 메타데이터로 포함하도록 요구하는 정책 강화(제안).
- 재현성 체크리스트(실험 환경·종속성·체크포인트) 의무화(제안).
(이들 비판·대안은 커뮤니티 토론 및 기존 재현성 연구에서 제기된 일반적 관점에 기초함 — (해석))

방법·재현성 체크리스트

  • 코드를 판정할 때 최소 확인 항목:
  • README에 ‘실험 재현 가이드’(데이터 경로·명령어 예시) 포함 여부.
  • 환경 명세(Conda/Pip/Container) 또는 requirements.txt/ Dockerfile 존재 여부.
  • 체크포인트(Weights) 제공 여부 및 다운로드 방법.
  • 라이선스 파일(LICENSE) 존재 여부 및 유형(예: MIT, Apache-2.0, CC-BY 등).
  • 논문 본문/보조자료에 코드 링크가 명시되어 있는지 교차검증.
    (이 체크리스트를 기준으로 자동화 크롤러를 설계하면 ‘code_share’ 메트릭의 신뢰도를 높일 수 있음 — 제안)

범위·한계 요약

  • 본 보고서는 CVF Open Access와 공개 GitHub 저장소(커뮤니티 집계·예시 레포)를 기반으로 한 질적·정성적 리뷰이다.
  • 정량적 통계(예: 전체 논문 대비 코드 공개 비율)는 본문에서 샘플링·자동화 크롤 검증을 수행하지 않았으므로 제공하지 않았다. 추가로 다음 작업이 필요하다: (1) 전체 논문 목록 크롤, (2) 논문 내 코드 링크 파싱, (3) 각 레포의 메타(라이선스·체크포인트 유무) 자동 추출 및 수동 샘플 검증.
